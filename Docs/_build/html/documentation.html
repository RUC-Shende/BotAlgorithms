Search-and-Exit Tourist and Superclass Method Documentation¶

This document aims to explain the methods and functions used in the superclass (iclData and iclVisual classes) as well as some non-API functions found in the Tourist class.

The use of global variables¶

Our original design of the algorithm simulator used way too many global variables to successfully keep developing on top of. The biggest reason for the switch to classes for all this information is that otherwise it would require us to essentially double the amount of global variables used to do things like comparison, and that just wasn’t a safe or efficient choice.

We do, however keep some global vars just because its easier. In the current implementation of superclass.js, we use a variable called superlist to keep track of all of our instances of iclVisual. All classes refer to superlist, so it is necessary for the simulation to work. We understand that this might not be the best practice, but is certainly better than before, and helps us meet the requirements to make this simulator work. Remember, we are essentially creating our own small game engine here with graphics, etc., and because Javascript is so weird about certain things (for example setInterval()’s context) we have to adapt to JS’s way of working.

To help implement comparisons and make switching between indices in the superlist easier, we create the controllerID which would just be the index of the current iclVisual we want to modify. When implementing comparisons we will likely just be switch between 1 and 0 for the controllerID the whole time.

This simulation is built on a mechanism that Javascript provides called an Interval. When we refer to our own Interval that makes the thing work, we call it theMotor. We are still testing comparisons, so there could be a chance of adding a second motor that focuses just on the second comparison, but we have to see how they play together before making that decision. The whole reason for the use of these global variables is because of this Interval, when a function is called on an interval, even if the method that is being called is specific to an instance of a class, the method’s reference for this change from the usual reference to the instance to a reference to the window. So, we have to use window variables to get what we want.

Other global variables that aren’t nearly as vital include a copy of instruBinder and algName that are changed depending on the current algorithm we are loading in, and unit2Px and center, which will be defined below as they relate to iclData.

The Tourist¶

A tourist in this case is the same exact thing as saying a ‘robot’ or ‘mobile agent’. It’s just a name we’ve had for the little circles in the sim since the beginning.

class tourist.Tourist(iclData, p)¶
Arguments:
  • iclData (iclData) – Represents the data of the overall simulation, robots’ tourlines and graphlines.
  • p (boolean) – True if bot is Priority Agents, false otherwise.
tourist.Tourist.DirectTo(value)¶

Send the robot to its next location (next frame) based on its current instruction or allowance.

Arguments:
  • value (Array) – [x,y] coordinates of the robot’s next calculated location.
tourist.Tourist.FollowWall(value)¶

A robot will follow the perimeter for a certain number of seconds, or indefinitely if no time specified. value[0] Will default to ‘right’ if no direction specified. Robot is required to be at a position on the perimeter.

Arguments:
  • value (Array) – [Direction string ‘left’ or ‘right’, <Optional>time to follow wall in seconds]
tourist.Tourist.GoOutAtAngle(value)¶

The robot will go from the interior of the object to an angle on the perimeter. Robot is required to be in the interior (not in perimeter) of the object.

Arguments:
  • value (Array) – [angle on the shape of the perimeter, percent distance of a radius]
tourist.Tourist.GoToCenter(value)¶

The robot will go to the center (origin) of the shape. Robot is not required to be at any specific position.

Arguments:
  • value (null) – null
tourist.Tourist.GoToExit(value)¶

Robot will go to the exit at cartestian coordinates (x, y). This is not the same as exitAngle.

Arguments:
  • value (Array) – [float x position, float y position]
tourist.Tourist.GoToPoint(value)¶

The robot will go to a point on the shape defined by cartesian coordinates (x, y) Based on original center point defined in this.iclData (before visual scaling)

Arguments:
  • value (Array) – [float x position, float y position]
tourist.Tourist.GoToWallAtAngle(value)¶

Makes a robot go to the wall at a specific angle on the shape.

Arguments:
  • value (integer) – Angle on the shape of the perimeter
tourist.Tourist.Intercept(value)¶

Robot will calculate the shortest to path to the closest targetable robot and create a straight path to intercept it. If there exist Priority Agents, the robot following this command will then target the closest of them. In case we ever use face-to-face priority algorithms.

Arguments:
  • value (null) – null
tourist.Tourist.Pursue(value)¶

Robot will continuously move in the direction of the closest target, until it catches it.

Arguments:
  • value (null) – null
tourist.Tourist.Wait(value)¶

A robot will wait for a time in seconds, or indefinitely if no time is specified.

Arguments:
  • value (float) – Time to wait for in seconds.
tourist.Tourist.WaitAverage(value)¶

A robot will wait at the average position of all robots NOT doing the WaitAverage command.

Arguments:
  • value (null) – null
tourist.Tourist.WallAtAngle(angle)¶

Calculates the wall position on the shape, given an angle.

Arguments:
  • angle (float) – Used to calculate the wall position of the shape.
Returns:

Array – Array with x,y position of the wall relative to the shape.

tourist.Tourist.knows¶

ICLData¶

This is a data structure that we use to maintain positions of Tourists and other relevant data when running the sim. We tried to keep it as removed from visuals as possible, data-only. ICLVisual uses this a lot when running the actual simulation.

class superclass.iclData(id, instruBinder, algorithmName, angle, wireless)¶

Data necessary to run a simulation of a given algorithm, create visual paths, etc. Tourist == Agent == Robot.

Arguments:
  • id (integer) – Global identifier of this object.
  • instruBinder (Array) – Array of valid trajectories for Tourists to follow. Length of this is touristNum.
  • algorithmName (string) – Full clean name of the algorithm instruBinder is supposed to represent.
  • angle (float) – Angle of the exit location on the shape. Exit location will be at the wall of the shape.
  • wireless (boolean) – True if using Wireless communication model, false otherwise.4
superclass.iclData.AllAtExit()¶

Check to see that the exit conditions have been satisfied. If so, modify this.time and this.timeMax to signal end of sim.

Called from this.AlterAnim(), Tourist.DirectTo(), Tourist.GoToExit().

superclass.iclData.LoadAnim()¶

Load and calculate the entirety of the animation, until time runs out. Used in the no-exit sim portion of this.Start(). Called from this.Start().

superclass.iclData.LoadPoints(i)¶

Create points {x:x’, y:y’}, so intercept function knows where a mobile agent will be. All agents will know others’ trajectories during the sim, this is a way of getting that info quickly. Used in the no-exit sim portion of this.Start(). Called from this.LoadAnim().

Arguments:
  • i (integer) – Which frame of the sim we will be calculating for.
superclass.iclData.Start()¶

Initializes the data structure, and runs a no-exit sim to initialize important values. Create and initialize all Tourists. Called from constructor.

superclass.iclData.algorithmName¶

superclass.iclData.allExitedLine¶

Whether or not all have exited. Can represent a visual.

superclass.iclData.center¶

superclass.iclData.degrees¶

Points of the equilateral shape to search. Ex. 3 = Triangle, 4 = square, 360 = circle

superclass.iclData.exitAlert¶

Someone learned where the exit is.

superclass.iclData.exitAllow¶

superclass.iclData.exitAngle¶

Angle of the exit to use for the sim.

superclass.iclData.exitFoundFrame¶

Frame on which the exit was found.

superclass.iclData.exitFoundLine¶

Whether or not the exit has been found yet. Can represent a visual.

superclass.iclData.fieldExit¶

Exit to use for the sim in {float}[x,y].

superclass.iclData.fps¶

Frames per second. Too high (> 100) == bad performance.

superclass.iclData.graphDots¶

superclass.iclData.graphLine¶

Array of touristNum strings, representing the current line left behind by a Tourist’s graph positions.

superclass.iclData.graphPoints¶

Array of Arrays of {x:, y:} objects to represent points on the graph. touristNum == number of inner arrays.

superclass.iclData.id¶

Global identifier to this data structure.

superclass.iclData.instruBinder¶

superclass.iclData.time¶

superclass.iclData.tourColors¶

Array of touristNum strings, representing the current line left behind by a Tourist’s trajectory.

superclass.iclData.tourPoints¶

Array of Arrays of {x:, y:} objects to represent points on the field. touristNum == number of inner arrays.

The ID given on creation is a nice way to ensure we are not mucking about in the data of another instance of iclData when using superlist[controllerID].

ICLVisual¶

This is the data structure which maintains all the visuals and updates the graphics on each frame. It does a mixture of modifying data in its own iclData member variable, and updating graphics based on them.

class superclass.iclVisual(iclData)¶

All visual updates and d3 calls for the sim happen in this. Rests on top of an instance of iclData. Field: The Shape, usually the left panel. Graph: The distance from exit graph, usually LoadPointsthe right panel. Multiple instances can and will be used for algorithm comparisons.

Arguments:
  • iclData (iclData) – The values necessary to update the visuals.
superclass.iclVisual.AlterAnim()¶

The main function used to control an algorithm when it is running for the first time. Recalculates all necessary points based on new exit location. Relies on window.superlist. This is due to the nature of calling AlterAnim from weird places, such as a time interval. Does both data calculation and visual updates at once, so while it does not really fit in either class, putting it in iclVisual was the best way to do it because this class rests on iclData. The two are really meant to always work together.

superclass.iclVisual.AlterLines(i)¶

Calculates the a given tourist’s current lines up to the current time. Called from this.AlterAnim().

Arguments:
  • i (integer) – The given tourist id for which to calculate the path.
superclass.iclVisual.ESlide()¶

Helper function for time slider. Disables the slider and shows help text again. Called by anonymous function inside LoadGraph().

superclass.iclVisual.Load()¶

Create tourist graph and field visuals Called by this.Start().

superclass.iclVisual.LoadField()¶

Create field visuals, including shape, reference angles and algorithm description. Called by this.Load().

superclass.iclVisual.LoadGraph()¶

Create graph visuals, including graph axes, labels, replay buttons, time/frame data and slider. Called by this.Load().

superclass.iclVisual.MSlide()¶

Helper function for time slider. Makes sure the slider stays within the bounds of timeMax. Called by anonymous function inside LoadGraph(). Relies on window.superlist.

superclass.iclVisual.PlayAnim()¶

The post-run algorithm controller. Allows for playback. timeDirect is defined in iclData. Relies on window.superlist because the call is usually from an interval once the sim has ended.

superclass.iclVisual.SSlide()¶

Helper function for time slider. Initializes the slider to be moved. Removes help text for visibility. Called by anonymous function inside LoadGraph().

superclass.iclVisual.Start()¶

Initializes the layers and ensures they take up the whole of the given SVG.

Called by constructor.

superclass.iclVisual.UpdateVisuals()¶

The last part that happens after data has been updated. Redraws tourist visuals, usually in relation to current time.

superclass.iclVisual.fieldSVG¶

A D3 reference to the Field. Contains 4 Layers - 1: BackGround, 2: Lines, 3: Bots, 4: OverLay

superclass.iclVisual.frameText¶

A D3 reference to the text which shows the current time in the sim (frames).

superclass.iclVisual.graphSVG¶

A D3 reference to the Graph. Contains 4 Layers - 1: BackGround, 2: Lines, 3: Bots, 4: OverLay

superclass.iclVisual.iclData¶

An instance of iclData to use to update visuals

superclass.iclVisual.lineFx¶

A reference to a D3 line function to create valid lines out of strings.

superclass.iclVisual.timeSlider¶

A D3 reference to the movable time slider that appears at the end of a normal sim.

superclass.iclVisual.timeText¶

A D3 reference to the text which shows the current time in the sim (seconds).

superclass.iclVisual.tourColors¶

An array of hex color codes for Tourists

Other Useful Functions¶

superclass.editAnims(s)¶

Outside controller for letting the buttons choose timeDirect. Relies on window.superlist. Called from the HTML <text>s relating to playback.

Arguments:
  • s (string) – The direction which to send the playback. Gotten from hardcoded method calls in HTML.
superclass.showAnims(s)¶

Unused function used to show two comparisons. Lets the user choose whether the want to see the graphs side by side, or the fields. Relies on window.superlist.

Arguments:
  • s (string) – The part of the animation to show side by side with another. Gotten from hardcoded method calls in HTML.
superclass.changeInstructions(n)¶

Unused function to swap instructions on the fly. Not really used anymore because we choose the algorithm based on the current #value in the URL, but I’ll leave it here just in case. Works best when having the user select an alg from a <select> HTML element.

Arguments:
  • n (string) – Algorithm Shortname. Will be defined in the docs… somewhere.
superclass.ChoosExit()¶

Unused function which lets a user choose the exit with a mouse.

superclass.exitPreview()¶

Gets the current value of an <input type=’number’> HTML element and displays the exit at that location for a preview. Relies on window.superlist. Reuses code from Tourist.DirectTo().

superclass.exitChosen()¶

Starts up the interval after clearing the field and graph of debris. Any new elements that need to be cleared after a run and before a new one… do it here. When doing comparisons the ‘theMotor = … ‘ is the part we wanna change to use superlist and controllerID to update both one after another. Can’t quite get it yet. Relies on window.superlist.

superclass.showHelp(s)¶

Select an algorithm or element description box to show based on current algorithm shortname.

Arguments:
  • s (string) – The id of the box to show. “steps” if you want to select an algorithm desc. box you’ve created based on shortname.
superclass.showAlgorithmDesc(s, w)¶

Contains the commands for all the algorithms. Give it a shortname and wireless true/false and it’ll change the instrubinder to match. Used on page load. The color isn’t really necessary.

Arguments:
  • s (string) – Algorithm shortname. Add ‘wl’ to shortnames where the algorithm has the potential to be f2f.
  • w (boolean) – Wireless true/false. A legacy value from before shortnames were more specific.
